# Install Nazar (Guided)

Use this skill when the user is setting up Nazar on a fresh MicroOS machine or reconfiguring an existing install. You are a conversational setup assistant — no TUI or wizard, just guided conversation.

## First-Run Detection

Check whether setup has already been completed:

```bash
[ -f /etc/nazar/.setup-complete ] && echo "already configured" || echo "fresh install"
```

If already configured, ask the user whether they want to reconfigure or exit.

## Distribution Model

Nazar runs on openSUSE MicroOS. The system is configured via `/etc/nazar/nazar.yaml` and deployed using Podman Quadlet containers generated by `nazar-setup.sh`. Users clone the repo and run setup:

```
~/Nazar/
  nazar.yaml              # main configuration (modules, identity, services)
  scripts/nazar-setup.sh  # generates Quadlet files from nazar.yaml
  scripts/nazar-object.sh # object store management
```

## Goals

1. Prevent disk UUID mismatch at boot.
2. Prevent password/login surprises by reusing the existing installer user.
3. Apply Nazar only after explicit user confirmation.
4. Configure optional modules via `nazar.yaml` enable flags.

## Prerequisites

- openSUSE MicroOS installed (KIWI image or standard installer)
- `transactional-update` available for system changes
- `podman` installed for container workloads

## Guided Flow

### Phase 1: Detect Environment

Gather system facts automatically — do not ask the user for these:

```bash
hostname
whoami
[ -d /sys/firmware/efi ] && echo "UEFI" || echo "BIOS"
uname -m   # x86_64 or aarch64
pwd
```

Determine:

- **hostname** — current machine hostname
- **username** — current user (or `$SUDO_USER` if running as root)
- **boot mode** — UEFI or BIOS (drives bootloader config)
- **architecture** — `x86_64` or `aarch64`
- **repo root** — current working directory (should contain `nazar.yaml` or be the target dir)

Tell the user what you detected:

> "I detected hostname **X**, user **Y**, boot mode **Z**, architecture **A**. I'll use these as defaults."

### Phase 2: Gather Configuration

Ask the user conversationally about each section. Use the detected values as defaults.

#### Identity

- **Hostname** — default: detected. "What hostname should this machine use?"
- **Username** — default: detected. "What Linux username should be the primary user?"
- **Timezone** — default: `UTC`. "What timezone? (e.g. Europe/London, America/New_York)"

#### Boot Loader

Auto-configure based on Phase 1 detection:

- **UEFI**: systemd-boot
- **BIOS**: GRUB + ask for boot device (default `/dev/sda`)

Tell the user what you'll configure and confirm.

#### Modules

Present the available modules with recommended defaults:

| Module           | Default | Description                      |
| ---------------- | ------- | -------------------------------- |
| `tailscale`      | on      | VPN for secure remote access     |
| `syncthing`      | on      | File synchronization             |
| `ttyd`           | on      | Web terminal (Tailscale-only)    |
| `desktop`        | on      | GNOME desktop + VS Code          |
| `passwordPolicy` | on      | Password strength enforcement    |
| `objects`        | on      | Object store data directory      |
| `heartbeat`      | off     | Periodic agent observation cycle |
| `matrix`         | off     | Matrix messaging channel         |

Ask: "Which modules would you like to change from these defaults?"

### Phase 3: Detect Hardware

Run hardware detection:

```bash
# Detect display manager
systemctl is-active gdm sddm lightdm 2>/dev/null
# Detect filesystem layout
lsblk -f
# Detect btrfs subvolumes (MicroOS default)
btrfs subvolume list / 2>/dev/null
```

If a display manager is active, note it — the user may want `desktop: false` to preserve their existing desktop setup.

Show the user a summary of the hardware detection and confirm.

### Phase 4: Generate Config Files

Generate `/etc/nazar/nazar.yaml` with the user's values substituted:

```yaml
# Nazar Configuration
identity:
  hostname: "<HOSTNAME>"
  username: "<USERNAME>"
  timezone: "<TIMEZONE>"

modules:
  tailscale: true
  syncthing: true
  ttyd: true
  desktop: true
  passwordPolicy: true
  objects: true
  heartbeat: false
  matrix: false

# repoRoot: "<REPO_ROOT>"  # uncomment if not ~/Nazar
```

If the current working directory is `/home/<USERNAME>/Nazar`, leave `repoRoot` commented out (it matches the default). Otherwise, uncomment it and set it to the actual `pwd`.

Show the user the generated config and ask for confirmation before writing.

### Phase 5: Write Files + Initialize

After user confirmation:

1. Write `nazar.yaml` to `/etc/nazar/`.
2. Initialize git repo and stage files:
   ```bash
   git init && git add -A && git commit -m "Initial Nazar configuration"
   ```

### Phase 6: Apply

Run the setup and apply:

```bash
sudo bash scripts/nazar-setup.sh
```

This generates Podman Quadlet files from `nazar.yaml` and applies the configuration.

For system-level package changes:

```bash
sudo transactional-update pkg install <packages>
sudo reboot  # transactional-update changes apply after reboot
```

If the apply fails:

- Show the error output.
- MicroOS uses btrfs snapshots for rollback: `sudo transactional-update rollback`
- Help the user diagnose and fix the issue.

### Phase 7: Verify

1. Run the verification script:
   ```bash
   ./scripts/verify-nazar.sh
   ```
2. Run tests to confirm everything works:
   ```bash
   npm test && bash tests/shell/run.sh
   ```
3. Write the setup-complete marker:
   ```bash
   sudo install -d -m 0755 /etc/nazar
   sudo touch /etc/nazar/.setup-complete
   ```
4. Print a summary:
   ```
   Nazar Setup Complete
   ====================
   Hostname:     <hostname>
   User:         <username>
   Timezone:     <timezone>
   Boot mode:    <UEFI/BIOS>
   Modules:      <enabled list>
   Config dir:   /etc/nazar/
   ```
5. Remind the user:
   - "Run `nazar` to start your AI assistant."
   - "Run `nazar apply` to apply future config changes safely."
   - "Run `nazar --skill ./skills/matrix-setup/SKILL.md` to set up Matrix messaging."

## Module Configuration Reference

All module enable flags live in `/etc/nazar/nazar.yaml`:

```yaml
modules:
  tailscale: true
  syncthing: true
  ttyd: true
  desktop: true
  passwordPolicy: true
  objects: true
  heartbeat: false
  matrix: false
```

## Troubleshooting

### transactional-update fails

If `transactional-update` fails, check for snapshot issues:

```bash
sudo snapper list
sudo transactional-update rollback <snapshot-id>
```

### Quadlet service won't start

Check Podman Quadlet generation and service status:

```bash
systemctl --user list-units 'nazar-*'
journalctl --user -u nazar-<service> --no-pager -n 20
```

### Missing system packages

Install system packages via zypper inside a transactional-update shell:

```bash
sudo transactional-update shell
zypper install <package>
exit
sudo reboot
```

## Safety Notes

- Do not run destructive disk commands.
- Do not edit boot/disk config without showing diff and asking the user first.
- Keep changes minimal and declarative.
- Always show generated config files to the user before writing.
- Offer rollback via btrfs snapshots if the apply fails.
